Overview of the Movella DOT Device API (Getting Started)

Introduction

This document gives an introduction to the Movella DOT Device API. It serves as a starting point for software developers interested in assessing the basis of the SDK.

Set-up

As can be seen, the host application developer can choose to use a Python, Java, C#, C or C++ interface. The C++ interface is not available in compiled form but is provided as part of the SDK as source code that is incorporated in the C header files. This C++ interface code implements a convenience wrapper around the C API. This means that the developer does not have to deal with memory management (i.e. easy object-lifetime management) as the class implementation takes care of this.

The API comprises of two C-interface libraries that are supplied for MS Windows and Linux, 64 bit. These are:

  • XsTypes that contains generic types (vectors, matrices, quaternions, etc.) and some basic operations on those types.
  • Movella DOT PC SDK that contains the access to functionality as implemented in Movella DOT devices.

The Movella DOT Device API library depends on the XsTypes, while XsTypes is an independent library.

The remainder of this document focuses on the C++ interface of the Movella DOT Device API only. For examples which will show how to import and use the API with all the available interfaces have a look at the Code Examples page.

Important Classes of the C++ interface

The following four classes available in the C++ interface are classes that are extensively used by a host application. Knowing what these classes are about will give the developer a starting point for implementing a basic application. If you go through the examples you will find these classes are used throughout all the examples.

  • XsDotConnectionManager: The central class of the Movella DOT Device API. It handles the administration of devices and global settings. It keeps track of all opened devices as XsDotDevice objects. An application creates a single XsDotConnectionManager object at startup and will use it to access devices and their data.
  • XsDotDevice: Provides an interface to control and configure a device. It will also process data that arrives for the particular device.
  • XsDotCallback: Provides event handling. If the host application needs to receive events from MDDA a sub class of XsDotCallback is created that overrides the virtual event functions the application needs to handle. An instance of the subclass is registered with the XsDotConnectionManager object. More information regarding callback implementation can be found in the Callback System and Concurrency section.
  • XsPortInfo: Contains the info of a detected device during scanning. Most importantly, it contains the Bluetooth address which will be used to establish the connection.

Detailed Recommended Program Flow

An application that wants to process data from one or more Movella DOT device will roughly follow the following steps

  • Startup phase

    1. Initialize MDDA

      The application starts with the creation of an XsDotConnectionManager object using the XsDotConnectionManager::construct function.

    2. Setup callbacks

      With event based data handling the application is signaled with information. To enable this a callback object must be registered. The application creates an object derived from XsDotCallback and overrides functions of interest, for instance onAdvertisementFound or onLiveDataAvailable. The object is registered with MDDA using the XsDotConnectionManager::addCallbackHandler function.

    3. Discover devices

      The application starts the discovery of devices by calling XsDotConnectionManager::enableDeviceDetection. This enables Bluetooth device detection. The XsDotCallback::onAdvertisementFound callback will be used to signal new received advertisements.

    4. Open devices of interest

      The port information (XsPortInfo) obtained in the previous step is used to open devices of interest using the XsDotConnectionManager::openPort function.

    5. Configure measurement mode and start measurement

      If needed the application configures the connected devices. Through the XsDotConnectionManager::device function the application obtains XsDotDevice instances required to do the configuration. Typical configuration actions are setting sync settings or enabling measurement mode with a certain payload. Depending on what data you need from the Movella DOT you can choose different payloads. You can also request the list of available filter profiles using getAvailableFilterProfiles and read and change this with onboardFilterProfile and setOnboardFilterProfile respectively. More information on how to use these can be found in the function documentation or the examples. These can be configured by setting the desired XsPayloadMode by using startMeasurement with a certain payload. The device will then stream data in real-time containing the chosen payload. For more information on the available payload modes please refer to the page Movella DOT Payload Modes .

  • Processing phase

    1. Handle incoming data

      The onLiveDataAvailable function of the callback object registered in the startup phase will be called whenever new data is available from the device. The function gets a pointer to the newly available XsDataPacket as parameter. The XsDataPacket::deviceId function is used to determine from which device the packet is received. Refer to Callback System and Concurrency for a more in-depth view on event based data handling

    2. Check for completion

      The application checks the conditions required for process completion, these are application defined. If the conditions are met the Exit phase is entered

  • Exit phase

    1. Close open devices

      All open devices are closed. The application uses the XsDotConnectionManager::closePort function on each opened device to do this or the XsDotConnectionManager::close function to close all open devices.

    2. Clean up MDDA
      The application releases the XsDotConnectionManager object created at startup using the XsDotConnectionManager::destruct function.

High Level Overview with Callbacks

Callback System and Concurrency

The Movella DOT Device SDK implements a concurrent system. This means that:

  • The MDDA functions are implemented in a thread-safe way.
  • The MDDA framework supports asynchronous events.

To make the MDDA functions thread-safe, data manipulation is controlled tightly and care is taken that function calls are non-blocking. This last point is also achieved by caching all information that would be required from the device.

The asynchronous events are handled by implementing a callback object. This can be achieved through creating a class inherited from XsDotCallback that overrides the virtual event functions. The callback object is registered in the XsDotConnectionManager object. An example for device detection and it's related callback are illustrated in the figure below.

Example Callback Flow

Another example use of the callback system was described in Detailed Recommended Program Flow In this example the application starts the measurements at the devices using the XsDotDevice::startMeasurement function. From this point on the connected device will generate data. Data will be sent over its Bluetooth connection. Each XsDotDevice object has set up listeners to handle incoming payload data.

After the data is read and converted from incoming byte arrays, the worker thread will call the onLiveDataAvailable() callback function. This call will end up in the overloaded function of the callback object implemented by the application developer.

It is good practice to take any actions based on callbacks in a separate thread. This can be achieved by using an OS messaging mechanism to forward the event to the applications' main event loop. The callback function runs in the context of XsDotDevice worker thread and problems can occur if MDDA functions that change the state of the MDDA (such as openPort or close) or the application (such as showing a window) are called from within the callback handler. For example, most GUI-toolkits require that all operations are performed from a single thread. Also, any time-consuming processing (e.g. graphical display) that is performed inside the callback function will block the worker thread, which causes a delay in data-processing and must therefore be prevented.

Another important consideration is the data that is passed as parameter of the callback function. To prevent unnecessary copying and thereby reducing time spent in callback functions, the data is given as a pointer to an XsDataPacket maintained by the MDDA. For the application, this data is only valid within the scope of the callback function. The calling worker thread remains in control of the data and is responsible for cleaning it up when it is no longer necessary. The developer can create an effective implementation in which only the data is copied that is actually required by the application and only create a message to be sent to the application thread containing this specific data.
